b+树的介绍，使用场景和实现

B树和B+树

B树的示意图：

B+树的示意图：

B树和B+树的相同点：

(1)B树和B+树都是多路平衡树，每一个节点含有多个元素

(2)B树和B+树更加适合磁盘的存储结构，因为B树和B+树每个节点有多个元素。从而可以降低树的高度，减少磁盘的访问次数。

B树和B+树的不同点：

(1)B树每个节点的元素都会记录索引(key)和数据(data)。

(2)B+树的非叶子节点(所以节点)的元素不记录数据(data)，只有叶子节点的元素记录数据(data)。

(3)B+树的叶子节点之间采用链表结构，用指针连接起来，方便范围查找。

B+树相对于B树更加适合做数据库的索引

  • B+树读写磁盘的代价更低

B树的每个节点的元素都会记录索引(key)和数据(data)，而B+树的只有叶子节点的元素记录数据(data)；

那么在磁盘块大小相同的情况下，B+树的每个节点可以存放更多的元素；

一个节点包含的元素越多，那么树的整体高度就约低，从而IO次数也就越少。

  • B+树的查找更加的稳定

B+树的只有叶子节点的元素记录数据(data)，并且B+树是平衡树，所有的叶子节点的高度也是相同的；

任何关键字的查找都会从根节点到叶子节点，所有关键字查询的路径长度相同，所以每一个数据的查询效率相当。

  • B+树更加适合做范围查找。

B+树所有的数据(data)都在叶子节点上，并且叶子节点通过指针串联起来，方便对数据进行遍历和范围查找。

mysql中的B+树可以存放多少索引：

  • mysql中的页大小是16K

  • 对于一个索引节点，一个元素(关键字+指针)占用大概14字节，那么一页可以包含16k/14=1170个元素

    一个元素包含关键字和指针，关键字为bigint类型占用8字节，指针占用6字节，加起来就是14字节

  • 对于一个叶子节点，一个元素(关键字+数据)占用大概1k字节，那么一页可以包含16k/1k=16个元素

  • 假设mysql中，索引B+树的高度是2

    第一层是1个索引节点，第二层是1170个叶子节点，所以总共是有1170*16=18,720条记录(万级别)

  • mysql中，高度是3的B+树

    第一层呢是1个索引节点，第二层是1170个索引节点，第三层是1170 * 1170=1,368,900个叶子节点(百万级别的叶子节点)所以总共是1,368,900 * 16=21,902,400条记录(千万级别的数据量)

B+树的插入：

首先我们假设，B+树每个节点的最大元素个数是3；

当给一个节点插入一个元素后，元素的个数超过了最大值(3)的时候，我们就要对该节点进行拆分。

B+树的插入过程：

  1. 插入元素的索引分别是(1，2，3)

  • 插入元素的索引分别是(1，2，3)；
  • 插入后的节点元素没有大于3，所以插入完成。
  1. 插入元素的索引是4

  • 插入索引是4的元素后，插入节点node的元素个数>3，需要进行分裂 ;
  • 把插入节点node分裂成两个节点，node和split_node；
  • 因为插入节点node没有父节点，需要创建新的索引节点parent为node和split_node父节点；
  • 最后把node的最小索引(1)和split_node的最小索引(3)插入到新的父节点parent中。

(3)插入元素的索引是5

  • 插入索引是5的元素后，插入节点node的最大个数不大于3；插入完成。

(4)插入元素的索引是6

  • 插入索引是4的元素后，插入节点node的元素个数>3，需要进行分裂 ;
  • 把插入节点node分裂成两个节点，node和split_node；
  • 因为插入节点node有父节点parent，所以可以直接把split_node的最小索引(5)直接插入父节点parent；
  • 把split_node的最小索引(5)插入到父节点parent后，父节点parent的元素个数不大于3；插入完成。

(5)插入元素的索引是7

  • 插入索引是7的元素后，插入节点node的最大个数不大于3；插入完成。

(6)插入元素的索引是8

  • 插入索引是8的元素后，插入节点node的元素个数>3，需要进行分裂 ;
  • 把插入节点node分裂成两个节点，node和split_node；
  • 因为插入节点node有父节点parent，所以可以直接把split_node的最小索引(7)直接插入父节点parent；
  • 把split_node的最小索引(7)插入到父节点parent后，父节点parent的元素个数大于3，所以父节点parent需要进行分裂；
  • 把父节点parent分裂成两个节点，parent和split_parent；
  • 因为父节点parent没有祖先，需要创建新的索引节点ancetor为parent和split_parent的父节点；
  • 最后把parent的最小索引(1)和split_parent的最小索引(5)插入到新的父节点ancetor中。

B+树的插入流程：

  1. 通过索引key，找到叶子节点node。
  2. 把key和value(或者只是key)插入到该节点node。
  3. 判断该节点是不是已经满了；如果该节点没有满，则插入结束；如果该节点满了，就执行步骤(4)。
  4. 把插入节点node拆分成2个节点node和split_node；然后判断被拆分的节点，有没有父节点；如果没有父节点，执行步骤(5)；如果有父节点，执行步骤(6)。
  5. 被拆分的节点没有父节点，首先创建一个父节点；然后把拆分的两个节点node和split_node的的最小索引插入到父节点，然后插入结束。
  6. 被拆分的节点有父节点，首先根据node找到父节点；然后跳到步骤(1)，把split_node的最小索引插入到父节点，开始新一轮的插入过程。

B+树的删除：

首先我们假设，B+树每个节点的最大元素个数是3；

但删除一个节点node的元素后，node节点的元素的小于等于最大值的一半(3/2=1)的时候；

如果该节点的兄弟节点sibling的元素个数大于最大值的一半(3/2=1)；就把sibling的元素给node；

如果该节点的兄弟节点sibling的元素个数不大于最大值的一半(3/2=1)；就合并sibling和node。

B+树的删除过程

(1)删除元素的索引是9

  • 找到索引9所在的叶子节点node。
  • 删除节点node，索引是9的元素，发现该节点元素的数量大于一半(1)，删除结束。

(2)删除元素的索引是8

  • 找到索引8所在的叶子节点node。
  • 删除node，索引是8的元素，发现节点元素的数量不大于一半(1)，然后找到他的兄弟节点sibling
  • 兄弟节点sibling元素数量大于一半，就把sibling元素(6)转移给node节点，并修改父节点的索引(7->6)

(3)删除元素的索引是7

  • 找到索引7所在的叶子节点node。
  • 删除node，索引7是的元素，发现节点元素的数量不大于一半(1)，然后找到他的兄弟节点sibling
  • 兄弟节点sibling元素数量不大于一半(1)，合并sibling和node的元素，并删除他们父节点parent的索引(6)
  • 删除parent索引6的元素，发现parent的数量小于等于一半(1)，然后找到父亲节点的兄弟节点parent_sibling
  • 发现parent_sibling元素数量大于一半(1)，然后把parent_sibling元素(3)转移给node节点，并修改父节点的索引(5->3)
  1. 删除元素的索引是6

  • 找到索引6所在的叶子节点node。
  • 删除node，索引是6的元素，发现节点元素的数量不大于一半(1)，然后找到他的兄弟节点sibling
  • 兄弟节点sibling元素数量大于一半，就把sibling元素(4)转移给node节点，并修改父节点的索引(5->4)

(5)删除元素的索引是5

  • 找到索引5所在的叶子节点node。
  • 删除node，索引是5的元素，发现节点元素的数量不大于一半(1)，然后找到他的兄弟节点sibling
  • 兄弟节点sibling元素数量不大于一半(1)，合并sibling和node的元素，并删除他们父节点parent的索引(4)
  • 删除parent索引4的元素，发现parent的数量小于等于一半(1)，然后找到父亲节点的兄弟节点sibling_parent
  • 发现sibling_parent元素数量不大于一半(1)，合并parent和parent_sibling的元素，并删除他们的父节点ancetor的索引(3)

B+树的删除流程：

  1. 通过索引key，找到叶子节点node。
  2. 根据key删除该节点node的元素
  3. 判断删除后node元素数量，是否大于一半；如果大于一半，则删除结束；如果没有大于一半，则执行步骤(4)
  4. 找到删除节点(node)的兄弟节点(sibling)
  5. 判断兄弟节点(sibling)元素数量，是否大于一半；如果大于一半，则执行步骤(6)；如果不大于一半，则执行步骤(7)。
  6. 如果兄弟节点(sibling)元素数量大于一半；首先sibling中的一个元素转给node，然后修改父节点(parent)的索引，删除结束。
  7. 如果兄弟节点(sibling)元素数量不大于一半；首先合并node和sibling节点，然后找到他们的父节点(parent)，跳转到步骤(2)，删除父节点中被合并的索引，开始新一轮的删除流程。
